# 生产运行手册

## 1. 支持边界

第一版生产拓扑为单台云服务器、单个 API 进程和单个 SQLite 数据库。Nginx/HTTPS、定时清理和系统级备份由部署者管理，不属于应用内功能。禁止同时启动两个指向同一 `DATABASE_PATH` 的 API 进程；SQLite WAL 可以改善读写并发，但不能把本系统变成多实例服务。

前端静态文件位于 `apps/web/dist`，API 监听 `HOST:PORT`。生产反向代理应让浏览器通过同一站点访问前端与 `/api`，避免额外 CORS 面和跨站 Cookie。`NODE_ENV=production` 时会启用 Secure 会话 Cookie，因此上游必须提供 HTTPS。

## 2. 目录与权限

建议将以下目录放在同一受保护的数据卷，并只授予运行 EDC 的系统账号读写权限：

| 配置              | 内容                                 | 备份要求                     |
| ----------------- | ------------------------------------ | ---------------------------- |
| `DATABASE_PATH`   | SQLite 主库及同目录的 `-wal`、`-shm` | 必须一致性备份               |
| `UPLOAD_ROOT`     | 受试者上传文件                       | 必须与数据库同一恢复点       |
| `QUARANTINE_ROOT` | 物理删除协调的临时隔离文件           | 建议备份，启动恢复会处理残留 |
| `EXPORT_ROOT`     | 可重新生成的导出文件                 | 可选                         |

不得把数据库、上传目录、导出目录或 `.env` 提交到 Git。运行账号不应拥有项目源码之外的额外系统权限。

## 3. 环境变量

以 `.env.example` 为模板，通过进程管理器注入生产值，不要把密钥写入源码。

```text
NODE_ENV=production
HOST=127.0.0.1
PORT=3000
DATABASE_PATH=D:/edc-data/edc.sqlite
UPLOAD_ROOT=D:/edc-data/uploads
QUARANTINE_ROOT=D:/edc-data/quarantine
EXPORT_ROOT=D:/edc-data/exports
SESSION_COOKIE_NAME=edc_session
SESSION_TTL_HOURS=12
TRUST_PROXY=true
```

- `TRUST_PROXY=true` 仅在 API 确实位于可信反向代理之后启用，否则客户端可伪造转发地址。
- `HOST` 推荐保持回环地址，只由本机反向代理访问。
- 所有数据路径应使用持久盘绝对路径；相对路径会相对于项目根目录解析。
- 同一站点只能使用一个稳定的 `SESSION_COOKIE_NAME`。

## 4. 首次部署

1. 在发布目录运行 `npm.cmd ci` 和 `npm.cmd run build`。
2. 创建数据目录并设置最小文件权限。
3. 注入环境变量。
4. 设置仅用于当前命令的 `EDC_ADMIN_PASSWORD`，运行：

   ```powershell
   npm.cmd run admin:init -- --username <用户名> --name "<显示名>"
   ```

   初始化命令会创建数据库、顺序执行全部迁移并创建唯一的新系统超级管理员；已有同名账号时拒绝覆盖。

5. 运行 `npm.cmd start`。启动时自动执行幂等迁移、恢复未完成的表单迁移与导出任务，并处理隔离目录残留。
6. 请求 `GET /api/health`，必须得到 `status: ok`。
7. 通过 HTTPS 登录，立即确认管理员账号、语言、主题和当前研究列表。

## 5. SQLite 运行参数

应用每次打开数据库时固定设置：

- `foreign_keys = ON`：启用外键约束。
- `journal_mode = WAL`：允许读取与单写入者并发。
- `busy_timeout = 5000`：锁竞争最多等待 5 秒。
- `synchronous = NORMAL`：在 WAL 模式下平衡持久性与吞吐。

生产约束：

- 只允许一个 API 实例、一个数据库文件和一个写入宿主机。
- 不得把 SQLite 文件放在 SMB/NFS 等网络共享目录。
- 不得在运行中直接复制单个 `.sqlite` 文件作为一致性备份。
- 磁盘必须监控剩余空间；数据库、WAL、上传和导出都会增长。
- 表单迁移和数据导出是持久化后台任务；进程收到正常关闭请求后会停止接收新请求，并等待已领取任务结束后再关闭数据库。
- CSV/Excel 仍由单实例进程生成。千行级任务已有自动化验收；更大的生产数据量必须在预发布环境按实际字段数压测，尤其关注 Excel 工作簿的峰值内存。
- 若出现长时间写锁或频繁 `SQLITE_BUSY`，应先停止扩容与写入，检查慢事务、磁盘状态和任务并发；本系统不支持通过增加 SQLite API 实例进行水平扩容。

## 6. 升级流程

1. 通知用户进入维护窗口，停止 API 进程；等待日志确认当前后台迁移/导出任务完成，并确认 3000 端口不再监听。不得强制终止正在写文件或切换表单版本的任务。
2. 对数据库、上传、隔离目录做同一恢复点备份；记录当前应用版本和 `PRAGMA user_version`。
3. 部署新源码并运行 `npm.cmd ci`、`npm.cmd run typecheck`、`npm.cmd run build`。
4. 使用与生产相同路径配置启动 `npm.cmd start`。迁移在监听端口前同步完成；任一迁移失败时进程不会正常提供服务。
5. 验证 `/api/health`、登录、研究列表、受试者详情、文件下载和审计查询。
6. 保留升级前备份，直到业务验收完成。

禁止在失败后把新版本程序连接到已部分人工修改的数据库。应先保留现场副本，再按恢复流程回到完整备份，定位迁移失败原因后重新升级。

## 7. 一致性备份与恢复

应用不提供内置备份按钮。推荐在停止 API 后执行冷备份：

1. 停止 API，确认没有进程持有数据库。
2. 将数据库文件、同目录可能存在的 `-wal`/`-shm`、`UPLOAD_ROOT` 和 `QUARANTINE_ROOT` 纳入同一个带时间戳的备份集。
3. 生成清单与哈希，记录应用版本、schema 版本、备份时间和目录路径。
4. 备份完成后再启动服务并检查健康端点。

恢复步骤：

1. 停止 API，并把当前故障现场整体移到独立保留目录，不要覆盖证据。
2. 将同一备份集中的数据库和文件目录恢复到新的空目录。
3. 校验备份清单、权限、路径及可用空间。
4. 使用备份时对应版本启动一次；确认数据可读后再执行目标版本升级。
5. 验证登录、跨中心数据范围、受试者数量、随机化结果、上传文件下载和审计记录。

数据库与上传文件必须恢复到同一时间点。只恢复其中一方会产生数据库路径存在但文件缺失，或磁盘孤儿文件。

## 8. 监控与故障处理

至少监控：进程存活、`/api/health`、HTTP 5xx、登录失败率、`SQLITE_BUSY`、磁盘空间、数据库/WAL 大小、上传与导出目录大小、备份成功时间。

日志中包含请求 ID，但不得主动记录密码、Cookie、CSRF token、文件正文或随机化未来序列。发生 500 错误时使用请求 ID 定位服务日志和对应审计事件。

## 9. 回滚边界

代码可以回滚，schema 不能假定向下兼容。数据库迁移仅向前执行，不提供自动降级。需要回滚时必须停止服务并恢复升级前的完整数据库与文件备份，然后启动与该备份匹配的旧应用版本。
