# 临床试验 EDC 系统实施计划

**状态：** 已确认并完成实施 v1  
**日期：** 2026-07-16  
**前置文档：** `DESIGN.md`、`design-system/clinical-trial-edc/MASTER.md`  
**执行状态：** 阶段 0–10 已完成，最终质量门禁已于 2026-07-17 通过

## 1. 实施目标与约束

本计划用于把已确认的产品设计落地为一个可在云服务器单实例运行的多项目、多中心 EDC 系统。

必须遵守：

1. 前端使用 Vue 3；PC/平板使用 Element Plus，手机专项页面使用 Vant。
2. 后端使用 Node.js，不引入其他服务端运行时。
3. 生产数据库仅使用 SQLite；数据库专属事务、锁和 SQL 语义集中在数据访问层。
4. 不全局安装任何软件或 npm 包；所有依赖只安装在项目目录，并在安装前取得用户同意。
5. HTTPS/Nginx、cron 备份、SDV、电子签名和离线录入不属于本次实现范围。
6. 先搭建全部页面路由和界面外壳，再逐模块完成真实业务能力。
7. 每个阶段必须通过相应测试与验收，不能只完成页面展示。

## 2. 总体架构决策

### 2.1 架构形态

采用单仓库、前后端分离、单进程 API 服务：

```text
浏览器（Vue 3 SPA）
        │ REST/JSON + Cookie Session
        ▼
Node.js Fastify API
  ├─ 身份认证与权限
  ├─ 项目/中心/成员
  ├─ 表单与数据记录
  ├─ 随机化
  ├─ 文件服务
  ├─ 导入导出
  └─ 审计与统计
        │
        ├─ SQLite 数据库（WAL）
        └─ 本地受控上传目录
```

不拆微服务，不引入 Redis、消息队列或对象存储。需要异步执行的表单迁移和大数据导出使用数据库持久化任务表，由同一 API 进程中的任务执行器处理。

### 2.2 选择 Fastify

选择 Fastify + TypeScript，而不是在第一版使用较重的企业框架：

- 插件边界清晰，适合按领域模块拆分。
- JSON Schema/OpenAPI 支持成熟，便于输入输出校验和 API 文档。
- 性能与启动成本适合单实例 SQLite 部署。
- 能显式控制事务、会话、文件和审计行为。

### 2.3 选择 Kysely 数据访问层

使用 Kysely 作为类型安全 SQL 查询构建器：

- SQLite 使用 `better-sqlite3` 驱动。
- 领域服务通过 Repository 或受控的数据访问函数操作数据库。
- SQLite 专属事务、锁和批量写入逻辑集中在数据库层与迁移脚本中。

不把动态表单数据绑定到 ORM 特有 JSON 查询，避免未来切换数据库时重写核心业务。

## 5. 前端应用架构

### 5.1 单应用、双组件体系

只建立一个 Vue 应用，共享路由、身份、权限、API、领域类型和动态表单内核。

- ≥1024px：默认桌面布局，Element Plus 侧栏/表格/弹窗。
- 768–1366px 平板范围：根据实际 CSS 视口和触控能力自适应，不依据设备型号或 User-Agent。
- <768px：移动布局，Vant 顶部栏、卡片、分步表单和底部导航。
- 768–1366px 区间同时存在平板和小型桌面设备，因此结合宽度、方向和 `pointer/hover` 媒体特征决定密度，而不是简单认定为某一机型。

复杂页面使用桌面/移动两个视图组件，共享 composable 与领域组件；简单页面只使用一套响应式模板。禁止同时渲染两套隐藏页面。

### 5.2 路由与页面外壳优先

第一批先建立全部已设计页面的真实路由、权限元数据、导航、面包屑、空状态和加载状态：

- 登录、项目列表、项目仪表盘。
- 受试者列表/详情、筛选、入组、数据录入。
- 表单列表/设计器/版本详情。
- 随机化方案/模拟/执行结果。
- 中心、成员权限、导出、审计和设置。
- 手机首页、受试者、新建筛选、随访和我的。

外壳阶段允许使用明确标记的 mock 数据，但 mock 必须集中在开发适配层，后续替换 API 时不得改写页面结构。

### 5.3 状态分层

- 服务器数据：模块 service + Pinia store，只缓存页面需要的数据。
- 会话状态：当前用户、当前项目、当前中心、权限集合。
- 界面状态：主题、语言、侧栏、筛选条件、未保存提示。
- 表单草稿：独立 store，以 `subject/form/visit/repeat` 作为草稿键。

项目或中心切换时清空相应业务缓存，并阻止在存在未保存表单时静默切换。

## 6. API 设计

### 6.1 约定

- 前缀：`/api/v1`。
- JSON 字段使用稳定英文 `camelCase`；界面文案由前端国际化处理。
- 错误响应统一包含 `code`、`message`、`details`、`requestId`。
- 列表接口统一使用分页、排序和结构化筛选。
- 所有写接口校验 `Content-Type`、CSRF token、项目上下文和权限。
- API 同步生成 OpenAPI 文档，作为联调契约。

### 6.2 模块路由

```text
/auth                 登录、退出、当前会话、修改密码
/users                全局账号与状态
/studies              项目及项目切换
/studies/:id/sites    中心
/studies/:id/members  成员、中心范围和权限
/studies/:id/forms    表单、版本、发布、迁移、导入导出
/studies/:id/visits   访视定义
/studies/:id/subjects 筛选、入组、状态和受试者详情
/subjects/:id/records 表单记录、重复记录和文件
/studies/:id/randomization 方案、模拟、启用和执行
/studies/:id/dashboard     指标与图表数据
/studies/:id/exports       数据导出任务
/studies/:id/audit-events  审计查询与导出
/settings             个人/系统设置
```

关键写操作使用幂等键：入组、随机化、文件提交和大数据导出。客户端重试不得创建重复受试者、随机号或文件记录。

## 7. 身份认证与权限实现

### 7.1 登录与会话

- 用户名/密码登录，不提供公开注册。
- 密码使用 Argon2id 哈希并保存独立 salt/参数信息。
- 登录成功生成高熵随机会话 ID；浏览器只保存 HttpOnly Cookie，数据库只保存会话 ID 哈希。
- Cookie 使用 `HttpOnly`、`SameSite=Lax`；生产环境在 Nginx HTTPS 后启用 `Secure`。
- 写请求使用 CSRF token；登录和敏感操作应用速率限制。
- 登录失败累计、临时锁定、账号禁用、主动退出和管理员重置密码后撤销相关会话。

### 7.2 首个管理员

提供本地初始化命令：

```text
npm.cmd run admin:init -- --username <name>
```

密码通过交互式隐藏输入或受控环境变量提供，不接受命令行明文密码。命令可重复执行但不能覆盖已有管理员。

### 7.3 权限计算

权限由以下集合计算：

```text
角色默认权限
+ 成员显式允许
- 成员显式拒绝
∩ 当前项目/中心作用域
```

显式拒绝优先于允许。后端每个 service 方法再次检查权限，不能只依赖前端路由守卫。Repository 查询必须带 `study_id`，中心级查询还必须带允许的 `site_id` 集合。

## 9. 动态表单引擎

### 9.1 共享 Schema

表单定义在 `packages/contracts` 中形成版本化 schema，设计器、运行时渲染器、后端校验和导入导出共用同一契约。前端校验只是即时反馈，后端必须使用相同规则重新验证。

字段使用永久稳定的 `fieldKey`：改标签不改 key，删除后 key 永久保留，不能被新字段复用。

### 9.2 设计器

实施顺序：

1. 字段组件库、画布和属性面板。
2. 字段排序、分组、复制和删除。
3. 基础校验和选项编辑。
4. 条件显示与字段间逻辑。
5. 计算字段表达式。
6. 重复录入和访视绑定。
7. 预览、发布、复制和导入导出。

拖拽之外必须提供上移、下移和移动到分组按钮，确保键盘操作与可访问性。

### 9.3 发布校验

发布前执行：

- 字段 key 与选项值唯一性。
- 条件依赖和计算依赖的循环检测。
- 引用字段存在性与类型兼容性。
- 必填字段在条件隐藏时的可达性。
- 访视与重复录入配置冲突检查。
- 与上一版本的兼容性差异分析。

### 9.4 自动迁移

新版本发布流程：

1. 创建候选版本和迁移计划。
2. 锁定该表单的新提交，现有页面提示稍后重试。
3. 持久化迁移任务，按批次处理记录。
4. 新字段补空；隐藏删除字段但保留值；执行允许的无损类型转换。
5. 每条记录更新表单版本并写审计事件。
6. 全部成功后激活新版本并解除锁定。
7. 失败时保持旧版本为活动版本，展示失败原因和可重试项。

任务可在进程重启后继续。迁移期间不允许出现部分用户使用新版本、部分用户使用旧版本的不可解释状态。

## 10. 受试者、访视与事件

### 10.1 编号服务

在事务内按项目维护筛选号、受试者编号和随机号序列。生成规则支持前缀和补零位数；号码一旦分配永不回收。

### 10.2 状态机

领域层实现显式状态机，拒绝非法跳转：

```text
筛选中 → 筛选失败
筛选中 → 待入组 → 已入组
已入组 → 已完成 / 已退出 / 失访
```

退出、失访或死亡不阻止获授权用户继续补录结局，但界面必须显示当前状态警告。

### 10.3 数据录入

- 单次表单使用唯一约束防止重复记录。
- 重复表单使用递增 `repeatIndex` 和记录日期。
- 访视表单关联 `subject_visit`。
- 长表单支持显式保存草稿和定时草稿保存。
- 提交采用乐观并发版本号；发现他人已修改时展示冲突，不静默覆盖。

## 11. 随机化实现

随机化算法放在无框架依赖的 `packages/domain` 中，通过固定输入得到可测试结果。种子和未来序列只允许随机化服务访问，不能通过普通 API 返回。

### 11.1 伪随机源

使用 Node.js `crypto` 构造基于种子、方案 ID、分层键和序列位置的确定性随机字节流。避免 `Math.random()`，确保可复现、可审计且不受进程重启影响。

### 11.2 算法

#### 简单随机化

按治疗组正整数权重进行加权抽取。每次分配推进不可回退的序列计数器。

#### 置换区组随机化

- 支持固定区组和多个可变区组大小。
- 区组大小必须能容纳分配比例的整数倍。
- 每个区组按比例生成治疗组多重集合，再用确定性随机源打乱。
- 未来区组不暴露给执行随机化的用户。

#### 分层区组随机化

- 对分层因素值生成规范化 `stratumKey`。
- 每个分层组合维护独立区组状态。
- 激活前计算理论分层组合数并给出过度分层警告。

#### 最小化随机

- 基于已完成分配计算各候选组的加权不平衡分数。
- 支持因素权重与随机成分概率。
- 最优组并非 100% 确定分配，避免完全可预测。
- 并列时使用确定性随机源打破平局。

### 11.3 冻结和并发

- 方案启用后可在首例前停用并修改；首例成功后立即冻结关键配置。
- SQLite 下随机化在 `BEGIN IMMEDIATE` 等价的独占写事务中完成。
- 事务内重新检查受试者未随机化、方案有效和幂等键。
- 依靠数据库唯一约束保证每名受试者最多一条成功分配。
- SQLite 使用短事务、唯一约束和单写入实例保证同等语义。

### 11.4 测试要求

- 固定种子的确定性快照。
- 各分配比例、多个治疗组和可变区组的不变量测试。
- 分层之间状态隔离测试。
- 最小化算法不平衡度和随机成分测试。
- 20 个以上并发请求对同一受试者随机化，必须只成功一次。
- 服务重启后序列连续性测试。

## 12. 文件上传与物理删除

### 12.1 上传

1. 流式接收文件并在写入时计算哈希。
2. 校验大小、扩展名和实际 MIME 类型。
3. 使用系统生成名写入临时目录。
4. 在数据库事务中创建元数据记录。
5. 原子移动到项目/中心/受试者路径。

文件下载只通过鉴权 API，设置安全的 `Content-Disposition` 和 MIME，不提供静态公开目录。

### 12.2 删除协调

文件系统和数据库无法共享事务，因此采用可恢复的隔离流程：

1. 将待删除文件原子移动到 `quarantine` 暂存目录。
2. 在数据库事务内写删除前审计快照并物理删除业务记录。
3. 事务成功后删除暂存文件；失败则移回原路径。
4. 启动时扫描未完成的删除任务并恢复或清理。

最终业务表和上传目录均不存在该文件；审计日志保留文件元数据但不保留正文。

## 13. 审计实现

所有写操作通过统一审计上下文执行，记录：请求 ID、操作者、项目、中心、对象、操作、前后快照、时间、IP/客户端摘要和原因。

- 业务 service 负责提供语义化操作名和快照。
- 数据写入与审计事件尽量处于同一数据库事务。
- 登录失败等无业务事务事件独立写入。
- 密码哈希、Cookie、CSRF token、随机化未来序列和文件正文必须从快照中过滤。
- 审计表无更新/删除业务接口。
- 导出审计日志本身也生成审计事件。

## 14. 仪表盘与导出

### 14.1 统计查询

按项目和允许中心范围聚合：筛选、入组、随机化、失败率、随访率、中心目标完成度和时间趋势。统计 service 不直接返回图表配置，只返回语义化数据，前端决定桌面/移动图表形式。

### 14.2 导出

- 小型 CSV 可流式同步下载。
- Excel 和大数据导出创建持久化任务，前端轮询状态。
- 用户选择项目、中心、受试者范围、表单、字段和数据格式。
- 导出任务保存请求时的权限范围；权限被撤销后未下载文件不再可访问。
- 临时导出文件设置到期清理策略。

## 15. 国际化、主题与可访问性

### 15.1 国际化

- 所有系统文案使用稳定 key，提供 `zh-CN` 和 `en-US`。
- 表单自定义标签原样展示，不自动翻译。
- 日期、数字和枚举显示使用 locale 格式器。
- 后端错误返回稳定 code，前端翻译用户可见消息。

### 15.2 主题

- 使用 CSS 语义变量映射 Element Plus 与 Vant 主题。
- 支持浅色、深色和跟随系统。
- ECharts 主题随应用主题切换并重新验证数据色对比度。
- 不在页面组件内散落原始十六进制颜色。

### 15.3 响应式与无障碍验收

- 测试 375、768、1024、1366、1440、1920px。
- 平板测试覆盖约 768–1366px CSS 视口、iPadOS/Android 类设备、横竖屏、触控和键鼠混合输入，不绑定 iPad Air 4。
- 全部触控目标至少 44×44px；手机输入框字号至少 16px。
- 表单标签、错误摘要、焦点管理、键盘操作和 `aria-live` 完整。
- 颜色不是唯一状态表达；正文对比度达到 WCAG AA。
- 支持 `prefers-reduced-motion`，无装饰性滚动动画。
- 手机不出现非预期横向滚动；宽表格改用卡片或专门的可控查看器。

## 16. 测试策略

### 16.1 测试金字塔

1. **领域单元测试：** 权限、状态机、编号、表单兼容、条件/计算和随机化算法。
2. **Repository 集成测试：** 使用临时 SQLite 数据库验证约束、事务和迁移。
3. **API 测试：** Fastify inject 验证认证、作用域、错误格式和审计。
4. **组件测试：** 表单字段、权限控件、主题和响应式组件。
5. **端到端测试：** Playwright 覆盖主要用户旅程。

### 16.2 必测用户旅程

- 初始化超级管理员并登录。
- 创建项目、中心和项目管理员。
- 创建筛选表单并发布。
- 新建筛选、手动判定、入组和生成编号。
- 配置并执行四类随机化方法。
- 创建访视表单、录入、修改并查看审计差异。
- 发布表单新版本并迁移旧记录。
- 重复录入不良事件并上传/删除文件。
- 中心用户无法查看其他中心；超级管理员可跨中心。
- 导出数据并产生审计事件。
- 中文/英文、浅色/深色和桌面/平板/手机切换。

### 16.3 数据隔离安全测试

对每个项目级 API 建立越权用例：

- 修改 URL 中的 study ID。
- 修改 subject/site/form ID 引用其他项目数据。
- 多中心用户访问未授权中心。
- 只读用户直接调用写 API。
- 已禁用用户复用旧会话。

测试必须证明后端拒绝访问，不能仅依靠前端隐藏按钮。

## 17. SQLite 生产边界

系统仅支持 SQLite，必须满足：

- 单 API 实例写入，启用 WAL、外键、忙等待和受控同步等级。
- 领域层不使用 `rowid`、隐式类型转换或无约束的跨项目查询。
- 主键统一使用 UUID，时间统一保存 ISO 8601 UTC 字符串，布尔值在 API 边界归一化。
- 编号、随机化和表单发布使用明确的短事务及唯一约束。
- 数据库迁移按 `user_version` 顺序执行并保留可重复的空库初始化测试。
- 自动化测试覆盖约束、事务、并发幂等、升级和后台任务恢复。

## 20. 主要风险与控制措施

| 风险                        | 控制措施                                             |
| --------------------------- | ---------------------------------------------------- |
| SQLite 多中心并发写入       | 单 API 实例、WAL、短事务、busy timeout、写入压力测试 |
| 动态表单升级破坏旧数据      | 稳定字段 key、兼容性分析、持久化迁移任务、审计快照   |
| 随机化重复或可预测          | 加密随机源、事务锁、唯一约束、不暴露未来序列         |
| 多项目/中心越权             | Repository 强制作用域、服务端权限、系统性越权测试    |
| 文件与数据库删除不一致      | quarantine 两阶段协调和启动恢复                      |
| Element Plus/Vant 风格割裂  | 共享语义 token、业务组件和统一交互规则               |
| 平板适配被误当单一机型      | 视口/输入能力响应式策略，多尺寸与横竖屏测试          |
| 审计日志快速增长            | 索引、分页、快照过滤；暂不自动清理                   |
| SQLite 容量或写并发达到上限 | 单实例写入、容量监控、定期归档并在扩容前重新评估架构 |
