# 临床试验 EDC 系统设计说明

**状态：** 设计草案 v1  
**日期：** 2026-07-16  
**范围：** 产品、权限、流程、信息架构、交互、视觉与领域数据设计；不包含技术实施步骤。

## 1. 产品目标

建立一个面向多项目、多中心临床研究的轻量 EDC 系统，使研究团队能够：

1. 独立管理多个研究项目及其中心、成员和权限。
2. 自定义筛选、基线、随访等电子表单。
3. 完成受试者筛选、入组和医学研究随机化。
4. 按访视或重复记录持续录入临床数据。
5. 查看入组/随访统计、导出数据并追踪全部数据变更。

### 1.1 MVP 范围

- 多研究项目与项目间数据隔离。
- 研究中心、账号、角色模板与细粒度权限矩阵。
- 自定义表单、条件显示、校验、计算字段、重复录入和访视时间点。
- 受试者筛选、入组、随机化、随访和全生命周期时间线。
- 仪表盘、CSV/Excel 导出、表单 JSON/Excel 导入导出。
- 中英文系统界面、浅色/深色主题、PC/iPad/手机适配。
- 全量审计日志和上传文件管理。

### 1.2 明确不做

- SDV、质疑管理、电子签名和法规认证。
- 测试/正式双环境工作流。
- 离线录入、扫码、自动随访提醒。
- 随机化撤销、重新随机或退出后名额释放。
- 系统内备份、Nginx/HTTPS 运维配置。
- 患者数据脱敏和导出脱敏。

## 2. 核心设计原则

1. **研究隔离：** 所有业务访问必须先确定项目上下文，再应用中心范围与权限。
2. **最小权限：** 角色提供默认权限，实际授权由细粒度权限矩阵控制。
3. **数据可追溯：** 新增、修改、删除、导入、导出、登录与随机化均留下审计记录。
4. **随机化不可逆：** 首例随机化后方案冻结；随机化成功后不可撤销或重做。
5. **表单可演进：** 发布版本不可就地修改，新版本通过兼容迁移保留历史数据。
6. **录入优先：** 界面追求清晰、稳定和高效，不用装饰干扰临床数据录入。
7. **跨设备一致：** 桌面承担配置和管理，手机聚焦筛选、查询与录入。

## 3. 组织、身份与数据作用域

### 3.1 组织层级

```text
系统
├─ 用户
└─ 研究项目 Study（数据隔离边界）
   ├─ 研究中心 Site
   ├─ 项目成员与权限
   ├─ 表单与访视计划
   ├─ 随机化方案
   └─ 受试者及其全部临床数据
```

一个用户可以加入多个项目，也可以在同一项目加入多个中心。权限按“用户 × 项目 × 中心（可选）”计算，不使用单一全局角色覆盖全部研究。

### 3.2 角色

| 角色              | 作用域         | 默认职责                                     |
| ----------------- | -------------- | -------------------------------------------- |
| 系统超级管理员    | 全系统         | 项目、全局用户、全项目数据与系统设置         |
| 研究项目管理员    | 单项目         | 项目配置、中心、成员、表单、随机化与数据管理 |
| 中心管理员        | 项目内指定中心 | 本中心成员与受试者管理，不可配置随机化方案   |
| 研究医生/研究护士 | 项目内指定中心 | 筛选、入组、随机化执行和获授权的数据录入     |
| 只读用户          | 项目或指定中心 | 查看获授权的数据和统计，不可修改             |

### 3.3 权限矩阵

角色是权限模板；项目管理员可在不突破自身权限的前提下为成员配置细粒度权限。建议权限键如下：

| 权限域   | 权限项                                       |
| -------- | -------------------------------------------- |
| 项目     | 查看、编辑基本信息、归档项目                 |
| 中心     | 查看、新增、编辑、停用                       |
| 成员     | 查看、邀请/创建、编辑权限、禁用、重置密码    |
| 表单     | 查看、创建、编辑草稿、发布、复制、导入、导出 |
| 受试者   | 查看、创建、编辑、删除、入组、退出/失访处理  |
| 随机化   | 查看方案、配置方案、执行随机化、查看结果     |
| 临床数据 | 查看、新增、修改、物理删除、上传/下载文件    |
| 数据导出 | 导出受试者、临床数据、统计数据               |
| 审计     | 查看审计日志、导出审计日志                   |
| 仪表盘   | 查看项目统计、查看跨中心统计                 |

权限判定顺序：账号状态 → 项目成员关系 → 中心作用域 → 权限项。超级管理员绕过项目/中心范围，但操作仍写入审计日志。

## 4. 研究项目设计

项目基本信息包括：研究名称、方案编号、申办方、研究类型、研究阶段、起止日期、默认语言、状态和备注。

项目状态：`草稿 → 进行中 → 已结束 → 已归档`。

- 草稿：配置中心、成员、表单与随机化方案。
- 进行中：允许筛选、入组、随机化与随访。
- 已结束：默认只读，可导出数据。
- 已归档：从日常列表隐藏，仅授权用户可查询。

每个中心包括名称、负责人、联系方式、入组目标数、状态。中心名称作为唯一标识，不设置中心编号或内部中心 ID。

## 5. 表单系统

### 5.1 表单类型

- 筛选表单
- 基线/入组表单
- 随访/访视表单
- 不良事件
- 合并用药
- 方案违背
- 终点事件
- 自定义表单

不良事件等是预设用途而非硬编码结构，项目管理员仍可自定义字段。

### 5.2 字段类型

支持单行文本、多行文本、数字、日期、日期时间、单选、多选、下拉选择、开关、文件、计算字段、量表、说明文本、分组标题。

字段通用属性：

- 唯一字段键、中文标签、帮助文本、占位提示、单位、默认值。
- 必填、只读、隐藏、是否纳入导出。
- 数值/文本/日期范围、正则规则和自定义错误提示。
- 单选/多选选项值与显示文本。
- 条件显示规则与字段间逻辑校验。

### 5.3 表单设计器布局

桌面采用三栏结构：

1. 左栏：字段组件库与表单结构树。
2. 中栏：表单画布、分组、字段排序和实时预览。
3. 右栏：选中字段/表单属性、校验和条件逻辑。

拖拽必须同时提供键盘可操作的“上移/下移/移动到分组”替代入口。手机不提供设计器。

### 5.4 条件与计算

条件规则使用可视化表达式：`字段 + 运算符 + 值`，支持 AND/OR 组。条件只控制显示与必填状态；被隐藏字段不参与当前校验，但已存在的历史值不会自动删除。

计算字段只能引用同一记录中已发布的数值/日期字段。计算结果只读，并保存计算所依据的表达式版本。

### 5.5 重复录入与访视

表单配置包含两个独立选项：

- **允许重复录入：** 同一受试者可创建多条记录，如不良事件、合并用药。
- **绑定访视时间点：** 项目管理员选择一个或多个自定义访视，如基线、7 天、30 天、3 月。

访视只描述计划结构，不实现自动到期提醒。受试者退出后仍允许授权用户继续录入结局数据。

### 5.6 草稿、发布与迁移

```text
创建/复制/导入 → 编辑草稿 → 预览与校验 → 发布 vN → 使用中
                                         └→ 创建下一版草稿 → 发布 vN+1
```

- 草稿可反复修改，不增加版本号。
- 已发布版本不可就地修改；新版本发布时自动递增。
- 新增字段：历史记录补充空值。
- 修改标签、帮助文本或兼容校验：直接使用新定义。
- 删除字段：新版界面隐藏，旧值保留并可在历史/导出中追踪。
- 删除选项：旧值保留，新记录不可再选择。
- 字段类型仅允许无损兼容转换；不兼容时阻止发布并说明原因。
- 每条数据记录保存当前表单版本号；迁移结果写入审计日志。

### 5.7 导入导出

- JSON：完整交换表单结构、规则和元数据。
- Excel：适合人工配置字段、选项和校验；复杂条件可使用规范化表达式列。
- 导入先进入预览页，展示新增、变更、冲突和错误；确认后生成草稿，不能直接覆盖已发布表单。

## 6. 受试者生命周期

### 6.1 编号

每名受试者具有：

- 筛选号：筛选创建时生成，项目内唯一。
- 受试者编号：确认入组时生成，项目内唯一。
- 随机号：成功随机化后生成，项目内唯一。

编号格式由项目配置前缀和补零位数；编号一经分配不重复使用。

### 6.2 状态

主状态为：

```text
筛选中 → 筛选失败
      └→ 待入组 → 已入组 → 已完成
                         ├→ 已退出
                         └→ 失访
```

随机化状态单独维护：`未随机化 / 已随机化`。不良事件、死亡、方案违背、合并用药和终点事件作为关联记录，不作为互斥主状态。

### 6.3 筛选与入组

1. 用户选择项目和中心，创建筛选记录。
2. 填写项目指定的筛选表单。
3. 系统执行字段校验，但不自动判断纳排标准。
4. 医生手动选择“符合入组”或“筛选失败”，并填写结论/原因。
5. 符合者确认入组，系统生成受试者编号和基线任务。

筛选、入组和随机化均使用确认页，明确展示项目、中心、患者标识和不可逆后果，避免在错误上下文中操作。

### 6.4 受试者详情

桌面详情页包含：概览、访视/表单、事件记录、文件、时间线、审计记录。顶部固定显示项目、中心、筛选号、受试者编号、随机号和当前状态。

时间线合并展示筛选、入组、随机化、访视、退出/失访、文件和数据修改，可按事件类型筛选。

## 7. 随机化设计

### 7.1 支持的方法

第一版提供医学研究常用方法：

1. **简单随机化：** 按分配比例独立抽取，适合样本量较大研究。
2. **置换区组随机化：** 固定或多个可变区组大小，维持阶段性平衡。
3. **分层区组随机化：** 按中心、性别、年龄层、疾病分期等因素形成分层后执行区组随机。
4. **最小化随机：** 根据已入组分布计算不平衡度，可配置随机成分，降低确定性。

所有方法支持两个或多个治疗组、任意正整数分配比例和自动/手动种子。中心可作为分层因素，但不能由中心管理员创建独立方案。

### 7.2 方案生命周期

```text
项目管理员创建草稿
→ 配置治疗组、比例、方法、分层因素、区组/最小化参数和种子
→ 模拟预览分布
→ 启用方案
→ 首例随机化后永久冻结
```

- 项目管理员和超级管理员可配置；中心管理员只能在有权限时执行。
- 启用前可修改，首例随机化后算法、组别、比例、种子和分层定义均不可修改。
- 随机序列/决策过程只在服务端产生，普通用户不能提前查看未来分配。
- 随机化成功即显示治疗组与随机号；不提供撤销、重做或名额释放。
- 同一受试者只能成功随机化一次；并发请求必须保证只有一次分配成功。

### 7.3 随机化记录

每次成功随机化记录：项目、中心、受试者、执行人、执行时间、方案及版本、方法、治疗组、分层值、种子引用、序列位置/决策摘要和随机号。审计界面不向普通录入用户暴露可推断未来分配的信息。

## 8. 文件设计

- 默认限制：单文件 20 MB；PDF、常用图片、Word 和 Excel。
- 目录按项目/中心/受试者/表单记录组织；数据库只保存相对路径与元数据。
- 文件名使用系统生成的不可预测标识，原始文件名作为元数据保留。
- 下载必须经过登录、项目、中心和权限校验，不能公开暴露存储路径。
- 删除临床记录时同步物理删除关联文件和数据库记录。
- 删除审计日志永久保留文件元数据、操作者和删除前业务快照，但不保留文件正文。

## 9. 审计与物理删除

审计事件至少包含：操作者、时间、请求标识、项目、中心、受试者、对象类型/ID、操作类型、变更前快照、变更后快照和原因（需要时）。

记录范围包括：登录成功/失败、账号与权限变动、项目/中心配置、表单发布与迁移、受试者/临床数据 CRUD、文件上传/删除、随机化、导入和导出。

- 业务实体允许按权限物理删除，但必须二次确认并填写原因。
- 普通录入用户默认无删除权限。
- 审计日志不可通过业务界面修改或删除。
- 审计日志中不保存密码、会话凭证或文件正文。

## 10. 信息架构

### 10.1 桌面端导航

```text
全局层
├─ 总览（超级管理员）
├─ 研究项目
├─ 全局用户（超级管理员）
└─ 系统设置

项目层
├─ 项目仪表盘
├─ 受试者
├─ 随访与数据录入
├─ 表单设计
├─ 随机化
├─ 中心管理
├─ 成员与权限
├─ 数据导出
├─ 审计日志
└─ 项目设置
```

顶部栏始终显示当前项目和中心作用域，支持项目切换、中心切换、主题、语言和个人菜单。不可访问的导航项显示禁用原因或按权限隐藏；涉及当前上下文的页面不能静默切换项目。

### 10.2 手机端导航

底部最多五项：`首页 / 受试者 / 新建筛选 / 随访 / 我的`。

手机不提供表单设计、中心/权限管理、随机化方案配置和批量导出。执行随机化入口位于受试者详情中，并需要独立确认。

## 12. 仪表盘与图表

默认指标：项目数、中心数、筛选数、入组数、随机化数、筛选失败率、随访完成率、中心入组目标与完成百分比、最近活动。

- 中心入组对比：横向条形图，显示目标线和完成数。
- 入组趋势：按周/月折线图。
- 受试者状态：不超过 5 类时使用环形图，否则使用条形图。
- 随访完成率：总体 KPI + 中心明细表。

图表均提供可访问的表格替代、无数据状态、加载失败重试和 CSV 导出入口。

## 13. 响应式与设备策略

| 设备                                           | 主要组件                | 行为                                             |
| ---------------------------------------------- | ----------------------- | ------------------------------------------------ |
| 1920×1080 PC                                   | Element Plus            | 高密度表格、侧栏、分栏设计器                     |
| 平板电脑（以 iPad Air 4 等常见设备为测试基准） | Element Plus 响应式布局 | 可折叠侧栏、触控友好尺寸、横竖屏，不绑定特定机型 |
| 手机                                           | Vant                    | 单列卡片、底部导航、分步表单、固定保存栏         |

平板适配面向一类设备而非 iPad Air 4 单一机型：覆盖常见 768–1366px CSS 视口、触控输入、横竖屏切换和 Safari/Chromium 浏览器。桌面表格在手机上不直接压缩；改为摘要卡片 + 详情页。手机长表单按分组折叠或分步展示，顶部显示进度，底部固定“保存草稿/提交”。不提供离线数据承诺；网络中断时保留当前页面内容并提示重试。

## 14. 国际化与主题

- 系统界面提供简体中文和英文，在个人设置中切换并记住偏好。
- 自定义表单内容由项目录入者维护，不强制同时填写中英双语。
- 日期、时间、数字按界面语言格式化，但数据库语义不因语言改变。
- 浅色/深色主题由语义 token 映射，不在页面中硬编码颜色。
- 主题可选浅色、深色、跟随系统。

## 15. 概念数据模型

```text
User ──< ProjectMembership >── Study ──< Site
                    └──< MembershipSite >── Site
                    └── Role + PermissionGrant

Study ──< Form ──< FormVersion ──< FormField
  │                    └── Condition / Validation / VisitBinding
  ├──< VisitDefinition
  ├──< RandomizationScheme ──< TreatmentArm
  └──< Subject ──< Screening / Enrollment / RandomizationAssignment
                 └──< DataRecord ──< DataValue
                                   └──< UploadedFile
                 └──< SubjectEvent

所有可变实体 ──< AuditEvent
```

所有项目业务表必须包含 `study_id`；中心范围数据同时包含唯一的 `site_name`。数据记录保存 `form_version_id`，确保表单升级后仍能解释历史值。

## 16. 关键空状态与错误状态

- 无项目：说明用途并提供“创建研究项目”。
- 项目无中心：引导先创建中心，阻止入组。
- 无已发布筛选表单：阻止新建筛选并给出配置入口。
- 随机化未配置/未启用：明确说明缺少的步骤。
- 权限不足：说明当前作用域和所需权限，不显示空白页面。
- 保存冲突：保留用户输入，提示数据已被他人修改并提供重新加载/对比。
- 网络失败：显示原因、重试入口和当前草稿保存状态。

## 17. 设计验收标准

- 用户能够从登录完成“进入项目 → 新建筛选 → 手动判定 → 入组 → 随机化 → 录入一次随访”的完整路径。
- 项目与中心作用域在所有数据页清晰可见，越权访问被阻止。
- 已发布表单升级不丢失旧字段值、不复用字段键、不产生无法解释的历史数据。
- 随机化方案首例后冻结，同一受试者无法获得两次分配。
- 任意新增、修改、删除、导入、导出和随机化都能在审计日志中定位。
- 375、768、1024、1440、1920px 均可完成其目标任务，无意外横向滚动。
- 浅色/深色主题正文达到 WCAG AA；所有关键状态同时具备文字或图标。

## 18. 已确认的基础设施边界

- 系统仅使用 SQLite 作为生产数据库，按单 API 实例、WAL 和短事务模式运行。
- 生产环境 HTTPS 由用户自行部署 Nginx 处理。
- 数据库和文件备份由用户后续通过 cron 实现，系统本身不提供备份界面。
- 首个超级管理员通过初始化命令创建，不开放公共注册。
